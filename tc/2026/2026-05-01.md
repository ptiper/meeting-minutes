## 1. Technical Decisions & Consensus

- **Topic:** RFC and Working Group Process Alignment
  - **Discussion:** Discussed the need to evolve the development process to handle the growing council. Proposed starting with a Request for Comments (RFC) for complex features before opening Pull Requests to resolve core debates upfront.
  - **Decision:** Approved adopting the new process. Working groups will handle deep dives, while the main TC meeting serves as a central point for status updates and critical decisions.

- **Topic:** Parallel and Stacked PRs
  - **Discussion:** Noted the difficulty of managing feedback and changes using private forks, especially regarding stacked PRs.
  - **Decision:** Approved adopting a pattern where core technical council members open parallel PRs stacked on top of the current branch in the main repository, paired with strict branch cleanup hygiene.

## 2. Review of open Working Draft PRs

- **Topic:** Identity Linking Scopes
  - **Discussion:** Discussed whether to define scopes per operation for checkout. The protocol needs to remain flexible to handle mandatory authentication, optional authentication for personalization, and no-authentication-required scenarios.
  - **Decision:** Agreed to define a minimum viable set of common scopes (e.g., `read` and `manage`) and avoid highly granular, operation-specific scopes. Suggested a deep dive to review the `required` field's role.
  - **Context:** [#354 feat!: identity linking OAuth 2.0 foundation with capability-driven scopes](https://github.com/Universal-Commerce-Protocol/ucp/pull/354)

- **Topic:** Loyalty Modeling and Balance
  - **Discussion:** Addressed whether the loyalty extension structure should adapt between enrollment and checkout, and how to model point redemption and balances.
  - **Decision:** Agreed that loyalty redemption will be modeled using payment handlers (aligning with split payments), while earnings forecasts remain in the loyalty extension. Loyalty balance will be modeled as a property of the instrument within payment semantics.
  - **Context:** [#340 feat: loyalty extension for checkout capability](https://github.com/Universal-Commerce-Protocol/ucp/pull/340)

- **Topic:** Attribution and Information Conveyance
  - **Discussion:** Discussed conveying attribution signals (like campaign parameters and deduplication keys) from consumer platforms to business backends. Evaluated whether this required formal protocol negotiation.
  - **Decision:** Reached consensus that attribution data does not require capability negotiation, as it is a platform-asserted, one-way communication of optional data.
  - **Context:** [#391 feat: attribution field for platform referral context](https://github.com/Universal-Commerce-Protocol/ucp/pull/391)
